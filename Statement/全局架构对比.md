## 一、项目目标（以实验方案文档为准）

### 核心流程

```
雷达持续采集
    │
    ▼
异常检测算法（实时分析雷达数据）
    │
    ├── 无异常 ──→ 系统静默，摄像头保持全局观察位
    │
    └── 检测到异常 ──→ 触发报警
                          │
                          ├─ 写 anomaly_events.csv
                          ├─ 终端打印报警
                          ├─ 下发云台指令：摄像头转向对应预置位
                          │      SC 搭靠 → 搭靠区预置位
                          │      KS 快速 → 快速通过区预置位
                          │      KA 靠岸 → 非正规靠岸区预置位
                          ├─ 截图/保存视频片段（取证）
                          └─ 云台回到全局观察位（等待下一次）
```

### 文档依据

| 原文 | 出处 |
|------|------|
| "相机用于完成**水池全局监控、异常区域转向和视频取证**" | 文档 P5 |
| "**系统识别到异常后**，控制电脑下发云台指令，使云台**自动转向对应区域**并保存视频或截图" | 文档 P9 |
| "云台相机提前设置对应预置位：搭靠区、快速通过区、非正规靠岸区。调用完毕后**恢复初始全局观察位**" | 文档 P9/P19 |
| "系统满足搭靠判定条件后**自动触发报警**，系统：'报警：船-船搭靠，**云台指令已下发**'" | 文档 P33 |
| "系统判断速度明显高于正常基准后，**自动触发报警**，系统：'报警：快速通过，**云台指令已下发**'" | 文档 P41 |
| "系统判断目标进入非正规靠岸区且出现减速停留后，**自动触发报警**……**云台指令已下发**" | 文档 P49 |

### 关键结论

**摄像头是被异常驱动的。没检测到异常 → 摄像头不动。检测到异常 → 摄像头转向对应预置位 → 取证 → 回全局位。**

## 二、当前生产代码的实际行为

### 当前流程

```
雷达数据
    │
    ▼
BoatTargetSelector._passes_filters() — 过滤噪声
    │
    ▼
AnomalyDetector.feed() — 检测三种异常
    │
    ├── 无异常 → 摄像头不动，系统静默
    │
    └── 检测到异常 →
            ├─ 写 anomaly_events.csv
            ├─ 终端打印报警
            ├─ BoatTargetSelector.select() 选最稳定的目标
            └─ 触发摄像头（预留接口，后续改为转向预置位）
```

### 与项目目标的差异

| 维度 | 项目目标 | 当前生产代码 | 一致？ |
|------|---------|-------------|--------|
| **摄像头平时状态** | 全局观察位，不动 | 异常触发后才动 | ✅ 一致 |
| **摄像头何时动** | 只有异常报警时 | 只有异常报警时 | ✅ 一致 |
| **choose_nearest** | 不需要 | 已替换为 BoatTargetSelector | ✅ 已修复 |
| **异常检测角色** | 核心调度者 | 先过滤噪声，再检测异常，再触发 | ✅ 一致 |
| **雷达 CSV 输出** | 持续记录 | 持续记录 | ✅ 一致 |
| **摄像头转向哪** | 三个预置位之一 | `on_anomaly()` 回调预留 | 🟡 待实现 |
| **基准测试** | 空场/正常航行/正规靠泊不触发报警 | 已支持（检测阈值可调） | ✅ 一致 |

### 已修复的差异

原来的 `choose_nearest` 每帧跟踪模式已替换为 `BoatTargetSelector` + 异常触发模式。摄像头不再每帧转动，只在异常检测到报警时动作。

## 三、原型代码分析

原型仓库 `test_3 - 副本` 包含三套独立的代码，做的事各不相同：

### 3.1 detect_anomalies.py — 离线异常检测

| 项目 | 说明 |
|------|------|
| **运行模式** | 离线批处理：读完整CSV → 一次性检测 → 输出事件CSV |
| **检测逻辑** | 搭靠/快速通过/靠岸，三种都有 |
| **摄像头控制** | **无**。纯算法，不管摄像头 |
| **与项目目标一致？** | ✅ 检测逻辑正确。但只有离线模式，缺少实时+摄像头联动 |

### 3.2 camera.py + tracker.py + main.py — 仿真跟踪系统

| 项目 | 说明 |
|------|------|
| **运行模式** | 仿真（matplotlib可视化），无真实硬件 |
| **逻辑** | 摄像头持续跟踪目标，预测式切换，跟踪质量评估 |
| **与项目目标一致？** | ❌ 这是持续跟踪模式，不是异常触发模式 |
| **与当前生产代码一致？** | ✅ 生产代码的 `live_tracking.py` 就是这套逻辑的真实硬件版 |

### 3.3 小结

```
原型仓库 = 两套独立系统：

  系统A: detect_anomalies.py       ← 异常检测（离线），逻辑正确，缺实时+摄像头联动
  系统B: camera.py + tracker.py    ← 持续跟踪（仿真），当前生产代码的前身

  项目需要 = 系统A的检测逻辑 + 实时化 + 异常触发摄像头（不是每帧跟踪）
```

    │           │
